# 电商购物车结算系统

## 1. 系统概述

### 1.1 功能描述

本系统是一个专注于后端逻辑的电商购物车结算引擎。它不处理用户注册或商品浏览，而是接收购物车状态（商品列表、数量）与营销规则（优惠券、运费相关规则），输出结算结果。系统可用于软件测试场景：提供**正确版基线实现**与**待测实现**，便于在同一输入下对比行为是否符合需求规格。

### 1.2 核心价值

- **逻辑复杂度高**：优惠券、运费、多商品组合等
- **边界条件丰富**：库存、门槛、地区、重量区间等
- **可测试性强**：输入输出为标准输入/标准输出文本，便于自动化脚本批量喂数与断言
- **对照测试友好**：同一请求可并行跑基线与待测版本

## 2. 核心模块设计

### 2.1 购物车管理模块

**功能**：维护购物车内的商品状态

- 支持添加商品（需校验商品是否存在）
- 支持移除商品
- 支持更新商品数量（需校验库存上限和最小购买量）

### 2.2 价格计算引擎

**功能**：计算商品小计、订单总价

- 基础定价逻辑：单价乘以数量
- 支持多种商品混合计算
- 处理价格精度问题

### 2.3 促销与优惠模块

**功能**：处理复杂的优惠叠加逻辑

- 支持"满减"规则（如满100减20）
- 支持"折扣"规则（如9折）
- 支持"一口价"规则
- 处理优惠券的互斥与叠加逻辑

### 2.4 输入读取与基础校验模块

**功能**：校验请求结构与业务约束，不满足时返回失败信息

- 从标准输入读取地区、商品列表、优惠券列表
- 对输入流进行基础有效性判断（如首段读取失败则退出）
- 当前待测实现未完整覆盖库存与字段合法性校验

### 2.5 结算输出模块

**功能**：生成结算结果文本

- 成功：输出单行 `status=SUCCESS final_payable=<数值>`
- 当前待测实现不输出失败结构体消息
- 便于自动化断言与回归测试

## 3. 需求规格说明书

### 3.1 业务规则

#### 3.1.1 商品规则

- 每个商品有唯一的SKU ID
- 商品数量必须为整数且大于0
- 商品单价精确到分（两位小数）
- 每个商品有库存量限制

#### 3.1.2 价格精度规则

- 所有金额计算保留两位小数
- 第三位小数采用"四舍五入"规则
- 特殊情况可采用"银行家舍入法"

#### 3.1.3 优惠叠加规则

**基本规则：**

- 系统支持多张优惠券
- 同类优惠券不可叠加（如两张满减券）
- 不同类优惠券可以叠加（如满减券+运费券）

**特殊规则：**

- 特价商品（标记为is_special=true）不参与任何满减活动
- 特价商品可享受折扣券
- 优惠券有使用门槛（如满200元可用）

#### 3.1.4 运费规则

**普通地区：**

- 首重 1kg 内 6 元
- 续重每 1kg 加 2 元（不足 1kg 按 1kg 计）

**偏远地区（新疆、西藏）：**

- 首重 1kg 内 15 元
- 续重每 1kg 加 10 元

**包邮规则：**

- 普通地区：订单在适用优惠后的商品金额满足门槛时可减免基础运费（具体门槛见实现）
- 偏远地区：按规则单独处理，不适用普通地区的包邮条件

#### 3.1.5 优惠处理规则（正确版基线）

- 商品行金额按单价与数量计算，并满足金额精度规则
- 支持多张优惠券；同类优惠券不可叠加
- 已过期或未达使用门槛的优惠券不参与结算
- 支持折扣、满减、一口价、运费券等类型（以接口约定为准）

#### 3.1.6 异常处理规则

- 当前待测实现主要执行计算流程，异常分支覆盖有限
- 输入首段读取失败时程序直接结束
- 库存不足、数量非法等规则在待测实现中未严格拦截

### 3.2 功能性需求

#### 3.2.1 基本功能

- 系统应能处理至少50种不同商品的混合结算
- 系统应支持多种优惠券类型及运费相关规则
- 系统应支持基线版本与待测版本的并行对照测试

#### 3.2.2 性能要求

- 系统应在50ms内完成一次结算计算
- 支持并发结算请求处理

#### 3.2.3 输入输出要求

- 输入采用标准输入文本协议（按空白分隔，支持换行）
- 输出为单行文本，字段包含 status 与 final_payable
- 金额字段按数值计算并直接输出

### 3.3 非功能性需求

#### 3.3.1 代码质量

- 代码应包含清晰的注释
- 复杂逻辑处需要详细注释说明
- 函数职责单一，易于维护

#### 3.3.2 安全性

- 输入数据需要进行有效性验证
- 防止SQL注入等安全漏洞
- 敏感信息需要加密处理

#### 3.3.3 可扩展性

- 系统设计应支持未来新增优惠类型
- 运费规则应易于扩展
- 支持配置化的业务规则

## 4. 接口定义

### 4.1 输入接口

#### 4.1.1 命令行输入接口

- 输入通道：标准输入（stdin）
- 解析方式：C++ 流式读取（cin >>）
- 分隔规则：空格、换行、Tab 均可作为分隔符

#### 4.1.2 请求示例

完整示例见 test_cases.json。核心格式如下：

```
region
item_count
sku_id name price quantity weight is_special stock
...（重复 item_count 行）

id type value min_purchase applicable_to_special expired
...（重复 coupon_count 行）

Xinjiang
2
SKU_001 MechanicalKeyboard 299 1 1.2 0 10
SKU_002 SpecialMousePad 9.9 2 0.1 1 5
1
CPN_100 discount 0.9 200 0 1
```

## 5. 测试用例示例

### 5.1 正常结算测试

**场景**：多商品订单并使用折扣券（见 test_cases.json 案例001）

**预期（待测实现）**：返回 status=SUCCESS，包含 final_payable 字段；数值应与 test_cases.json 中对应案例 output 一致。

### 5.2 优惠券规则测试

**场景**：同类券互斥、门槛、过期券

**预期**：同类优惠券不应重复生效；未达门槛或已过期的券不参与结算。

### 5.3 特殊商品测试

**场景**：订单同时包含普通商品与特价商品，并使用仅作用于普通商品的折扣策略

**预期**：折扣基数应符合 `applicable_to_special` 与业务规则约定。

### 5.4 偏远地区与运费测试

**场景**：收货地区为新疆或西藏，或商品总重量处于续重边界

**预期**：运费按偏远地区规则计算；续重进位规则符合规格。

### 5.5 异常与失败路径测试

**场景**：库存不足、负数量、门槛与过期券等规则偏差（见 test_cases.json 多案例）

**预期**：用于发现待测实现与业务规格的不一致，不以 FAIL 消息结构作为唯一判定条件。

## 6. 测试关注点建议

- **等价类与边界值**：价格、数量、重量、门槛金额、包邮条件附近
- **组合场景**：多商品 + 多券 + 运费券
- **对照测试**：同一输入下，基线实现与待测实现的输出差异分析
- **鲁棒性**：输入字段缺失、数量/金额异常、优惠券边界值

## 8. 总结

本电商购物车结算系统在需求层面强调规则清晰、输入输出结构化，当前待测程序采用标准输入文本协议与单行文本输出，适合通过脚本进行批量回归测试。通过提供基线实现与待测实现，可在固定需求规格下开展对照测试与缺陷分析。
